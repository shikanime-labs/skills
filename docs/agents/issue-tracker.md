# Issue Tracker

<!-- markdownlint-disable MD013 MD033 -->

Where work is tracked for this repo, and how the workflow skills read from and
write to it. `bootstrap` writes a repo-specific copy of this file; this is the
default convention.

## Tracker

**GitHub Issues** on the repo's `origin` remote.

Resolve owner/repo from the git remote:

```bash
REMOTE_URL=$(git remote get-url origin)
OWNER_REPO=$(echo "$REMOTE_URL" | sed -E 's|.*github\.com[:/]||; s|\.git$||')
OWNER=${OWNER_REPO%%/*}
REPO=${OWNER_REPO##*/}
```

## Auth

Prefer the GitHub MCP tools when available (`GitHub_issue_write`,
`GitHub_list_issues`, `GitHub_sub_issue_write`, ...). Otherwise fall back to the
`gh` CLI, then to `curl` against the REST API:

```bash
if command -v gh &>/dev/null && gh auth status &>/dev/null; then AUTH=gh
else AUTH=curl; fi   # curl needs $GITHUB_TOKEN
```

## Label vocabulary

Two **category** labels and five **state** labels drive triage. These are the
canonical role names; `bootstrap` maps each role to the label string that
**already exists** in the repo (read via `gh label list` / GitHub MCP). If a
repo has no label for a role, `bootstrap` proposes the default; it never creates
a duplicate under a different name.

| Canonical role  | Default label     | Meaning                                   |
| --------------- | ----------------- | ----------------------------------------- |
| bug             | `bug`             | Something is broken                       |
| enhancement     | `enhancement`     | New feature or improvement                |
| needs-triage    | `needs-triage`    | Maintainer must evaluate                  |
| needs-info      | `needs-info`      | Waiting on reporter                       |
| ready-for-agent | `ready-for-agent` | Fully specified, an AFK agent can take it |
| ready-for-human | `ready-for-human` | Needs a human                             |
| wontfix         | `wontfix`         | Will not be actioned                      |

The **default label** column is the string `bootstrap` proposes when a repo has
no label for that role. The actual string used in the repo is recorded in the
per-repo copy of this file (the `Label vocabulary` table gets rewritten with the
real names). Always reference labels through that mapping, never hardcode the
defaults.

Every triaged issue carries exactly one category label and one state label.

## Creating an issue

```bash
# gh
gh issue create --title "TITLE" --body "BODY" --label "enhancement,ready-for-agent"
# curl
curl -s -X POST -H "Authorization: token $GITHUB_TOKEN" \
  "https://api.github.com/repos/$OWNER/$REPO/issues" \
  -d '{"title":"TITLE","body":"BODY","labels":["enhancement","ready-for-agent"]}'
```

## Meta issues (strategic)

A **meta issue** carries strategic value — the spec/PRD from `to-spec` is the
canonical meta issue for a body of work (an epic). Break it into tactical
tickets via `to-tickets`, and **set each tactical ticket's parent to the meta
issue** so the work rolls up.

- The meta issue is strategic: label it `ready-for-human` (a human approves the
  direction before any code). Its tactical children carry `ready-for-agent` and
  drive the atomic PR stack.
- Create the parent link when publishing each ticket:
  - `gh issue create ... --parent <meta-issue-number>`
  - or `GitHub_sub_issue_write` (method: add, sub_issue_id: <ticket>,
    issue_number: <meta-issue>)

## Blocking edges (sub-issues / dependencies)

Tickets declare the tickets that **block** them. On GitHub, model this with
native sub-issues (`GitHub_sub_issue_write`) or a `Blocked by: #N` line in the
body when sub-issues aren't available. A ticket with no unresolved blockers can
be started immediately.

## AI disclaimer

Every comment or issue an agent posts during triage **must** start with:

```text
> *This was generated by AI during triage.*
```
